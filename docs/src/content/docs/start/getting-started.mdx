## Install

```bash
npm i prop-for-that
```

## Three usage modes

### 1. Auto (zero-config)

Import `prop-for-that/auto` and it wires everything for you — declaratively, with nothing attached unless the DOM asks for it:

- Scans the DOM for `[data-props-for]` attributes and binds the named sources to those elements. Globals are declared too — put them on the root `<html>` and they write to `:root`.
- **Loads plugin sources on demand.** The first time a `data-props-for` key needs a plugin, its chunk is dynamically imported and registered, then the binding attaches — no `registerPlugins()`, no separate plugins import.
- Keeps everything in sync with a single `MutationObserver`: elements added later are bound, removed elements are torn down and their properties cleaned up, and editing an element's `data-props-for` value re-syncs just the keys that changed — so you can add or drop a source at runtime by changing the attribute alone.

Because plugins lazy-load via dynamic `import()`, load `auto` as a **module** script (the bare `<script src>` drop-in is gone):

```html
<!-- from a CDN that serves the dist tree verbatim (unpkg / jsDelivr file paths) -->
<script type="module" src="https://unpkg.com/prop-for-that/dist/auto.js"></script>
<!-- or, with a bundler -->
<script type="module">import 'prop-for-that/auto'</script>

<html data-props-for="viewport pointer">   <!-- globals → :root -->

<input type="range" min="0" max="100" data-props-for="range">
<section data-props-for="size visibility">…</section>
```

```css
input {
  background: hsl(calc(var(--live-value-pct) * 120) 80% 50%);
}

section {
  opacity: var(--const-has-entered);
  transition: opacity 0.4s;
}
```

:::caution[Two limits of the auto path]
- **Shadow DOM is invisible.** The `MutationObserver` doesn't cross shadow boundaries, so `data-props-for` inside a web component's shadow root is never auto-bound. Bind those imperatively: `propsFor(shadowEl, ['…'])`.
- **CDN must serve raw files.** On-demand plugin loading uses dynamic `import()` of code-split chunks that ship next to `auto.js`, so use a CDN that serves the `dist` tree verbatim (unpkg, jsDelivr file paths). Bundling/rewriting CDNs (esm.sh, `?module`, jsDelivr `+esm`) can rewrite the entry and break lazy plugin loading.
:::

:::tip[One element, many sources]
`data-props-for` takes **one or more source keys, separated by spaces**. `data-props-for="size visibility"` binds *both* `size` and `visibility` to that element, and each writes its own `--live-*` / `--const-*` properties side by side:

```html
<article data-props-for="size visibility pointer-local">…</article>
```

```css
article {
  /* size → --live-w/--live-h, visibility → --const-has-entered,
     pointer-local → --live-local-pointer-x-ratio/--live-local-pointer-y-ratio — all on this one element */
  aspect-ratio: var(--live-w) / var(--live-h);
  opacity: var(--const-has-entered);
  background: radial-gradient(circle at calc(var(--live-local-pointer-x-ratio) * 100%) calc(var(--live-local-pointer-y-ratio) * 100%), white, transparent);
}
```

Order doesn't matter and duplicates are ignored. You can mix core and registered plugin keys. Because the list is re-synced whenever the attribute changes, adding or removing a key at runtime binds or tears down just that one source — the others keep running.
:::

Want the values to **interpolate**? Add `data-props-typed` to the root `<html>` and every `--live-*` this page writes is registered as an [`@property`](/concepts/typed-properties/) — the markup equivalent of `configure({ typed: true })`:

```html
<html data-props-typed>…</html>
```

### 2. Imperative

Import individual functions for explicit control over which sources are active and when they're torn down:

```js
import { propsFor, configure } from 'prop-for-that'

// write viewport to :root (pointer is now an opt-in plugin — see Plugins)
propsFor(['viewport'])

// write the range source to a specific element; returns a disposer.
// an element's id is a ready-made global in JS — no query selector needed.
// here, <input id="slider" type="range" min="0" max="100">
const dispose = propsFor(slider, ['range'])

// later, tear it down cleanly
dispose()
```

Use `configure()` to change the property prefixes or the global root target, before attaching any sources:

```js
import { configure } from 'prop-for-that'

configure({ livePrefix: '--pft-', constPrefix: '--pft-const-' })
```

### 3. Head constants

A few values are read once and matter for the first paint: scrollbar width, device pixel ratio, CPU cores, and the low-entropy user-agent bits (OS, browser, engine). Load the head entry as a synchronous inline script so the properties exist before any CSS applies:

```html
<head>
  <script type="module">import 'prop-for-that/head'</script>
  <!-- sets --const-scrollbar-w, --const-scrollbar-thin-w, --const-scrollbar-overlay,
       --const-dpr, --const-cores, --const-mem,
       --const-ua-platform, --const-ua-browser, --const-ua-engine, --const-ua-version, --const-ua-mobile -->
</head>
```

These use `--const-*` names and are never overwritten after that first write.
