title: Style queries
description: Use @container style() to apply conditional CSS rules from the discrete properties prop-for-that writes.
---

There are two ways CSS consumes the properties prop-for-that writes:

- **`var()`** — *interpolate* on a value. Best for **continuous** numbers (`--live-pointer-x-ratio`, `--live-value-pct`, `--live-visible-ratio`) fed through `calc()`.
- **`@container style()`** — apply a **conditional rule** when a property equals a value. Best for **discrete / boolean / enum** state (`--live-online`, `--live-value: 100`, `--live-paused`, `--live-battery-low`, capability tiers).

Style queries are what make the *discrete* signals shine: they let a value flip whole rule blocks on and off — something `var()` can't do.

## The container rule

A style query matches against an element's **nearest ancestor container** — and for style queries, *every element is a container by default* (you do **not** need `container-type`). So:

- **Global sources** write to `:root`, an ancestor of everything → queryable from anywhere.
- **Element sources** write to the bound element → its **descendants** can query it (an element can't style-query its own property, only an ancestor's). With the container-bound `range` pattern, that's exactly why a gauge *inside* the bound wrapper can react.

## Example 1 — a global boolean (offline)

```js
import { register, global } from 'prop-for-that'
import { online } from 'prop-for-that/plugins'

register(online)
global(['online'])   // writes --live-online (1/0) to :root
```

```css
.offline-banner { display: none; }

/* :root is an ancestor container, so this works anywhere on the page */
@container style(--live-online: 0) {
  .offline-banner { display: block; }
  body { filter: grayscale(1); }
}
```

## Example 2 — an element milestone (slider at max)

```js
bind(document.getElementById('meter'), ['range'])  // --live-value on #meter
```

```css
/* .gauge is inside #meter and inherits --live-value, so its descendants
   can match the exact integer — impossible to express with var()/calc() */
@container style(--live-value: 100) {
  .gauge__num { color: var(--accent); }
  .gauge__flag::after { content: 'max'; }
}
```

## Example 3 — a capability tier *(proposed `capability` plugin)*

```css
/* ship light media on lite connections */
@container style(--live-delivery-mode: 1) {
  .hero-video { display: none; }
  .hero-poster { display: block; }
}
```

## Equality today, ranges later

Style queries currently match on **equality** (`style(--live-net-type: 4)`), not comparisons. Range style queries (`style(--live-battery-level < 0.2)`) are **not yet stable** across browsers.

That's a feature, not a bug, for this library: emit **discrete booleans and tiers from JS** and you get threshold styling that works *today*. For example, the `battery` plugin's `--live-battery-low` (1/0) or a capability tier lets you write:

```css
@container style(--live-battery-low: 1) {
  .autoplay { animation-play-state: paused; }
}
```

…instead of waiting for range queries. Pair continuous values with `var()`, discrete values with `@container style()`.

## Browser support

Style queries for custom properties ship in Chromium 111+, Safari 18+, and Firefox 128+. Where unsupported, the `@container style()` blocks are simply ignored — so treat them as progressive enhancement and keep a sensible default outside the query (as the examples above do).
